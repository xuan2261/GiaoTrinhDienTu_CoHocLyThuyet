## Key Insights

- DeCuong works because interaction is obvious: drag the object/vector, read cards update.
- Current engine has route-owned handles; this phase makes that contract user-facing and testable.
- Controls must stay minimal. Sliders are secondary when direct drag exists.

## Requirements

### Functional

- Handle hit area >= 44px touch target without visually oversized markers.
- Drag feedback shows active object, ghost/guide, or short label.
- Readout cards update during drag, not only after release.
- Keyboard nudge remains available for focused handles.
- Reset restores initial state and clears transient trails.
- Animated routes expose play/pause and speed only where meaningful.
- Route controls use consistent types:
  - slider for scalar continuous values
  - segmented buttons for presets
  - toggle/select for modes
  - direct drag for geometry/vector state

### Non-Functional

- No global reset functions.
- No generic fallback handle for routes that have behavior-defined handles.
- No route loses touch support.

## Architecture

```text
Behavior handles
  -> SimInteractions createInteractionLayer
  -> active handle feedback
  -> state setter
  -> draw()
  -> renderReadoutCards()
  -> optional formula panel update
```

## Risk Assessment

| Risk | Mitigation |
|---|---|
| Handle changes break route math | Phase route groups verify physics formulas afterward |
| Hit radius causes accidental drags | Separate hit radius from visible marker; test mobile |
| Too many controls | Per-route controls limited by scene learning objective |

## Security Considerations

- No HTML injection in handle labels or readout values.
- Continue using `textContent`.
